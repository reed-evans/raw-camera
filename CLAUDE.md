# RAWCamera — CLAUDE.md

Offline iOS RAW / ProRAW manual camera. SwiftUI + AVFoundation + Metal + CoreMotion.
The product goal is a **pro manual camera that captures honest sensor data** (RAW/ProRAW
DNGs) with editorial "dark-luxury" instrument-style monitoring — not a point-and-shoot.
It deliberately does **no** computational photography (no Night mode / Deep Fusion /
multi-frame fusion); a capture is a single sensor readout. It is **fully offline** — no
networking, analytics, or telemetry of any kind (enforced by a gate).

## Architecture at a glance

Two targets, split for **verifiability**:

- **`CameraCore/`** — a pure-logic Swift Package (SPM). Imports **no Apple media framework**
  (no AVFoundation/Metal/CoreMotion/Photos). Holds clamp/white-balance/histogram/level/
  RAW-selection math and the frozen `PreviewUniforms` ABI. **Fully host-testable** (`swift test`,
  no camera). This is where logic belongs — push it down here and unit-test it.
- **`App/`** — the iOS app (AVFoundation capture, Metal preview, SwiftUI UI, CoreMotion). **Not**
  machine-verifiable for live capture — the final gate is a human on a device.

`CameraModel` (`@MainActor @Observable`) is the single UI-facing state object; the UI binds to
it and it depends on `any CameraCapturing` (the frozen protocol the capture stack conforms to).
Background callbacks (`onVideoFrame`, `onConfigured`, `onCaptureFinished`, motion, histogram)
fire off the main thread and hop to the main actor before touching observable state.

**The authoritative architecture/contract spec is [CONTRACTS.md](CONTRACTS.md)** — module
ownership, the `CameraCapturing` protocol, the `PreviewUniforms` frozen ABI, the `CameraModel`
surface, and the CameraCore pure-logic signatures. Read it before changing anything structural.
History/process: [BUILD-PLAN.md](BUILD-PLAN.md), [status.md](status.md). On-device acceptance:
[DEVICE-CHECKLIST.md](DEVICE-CHECKLIST.md).

## Build / test / verify — the critical workflow

The `.xcodeproj` is **generated by xcodegen and gitignored**. After editing `project.yml`, adding/
removing files, or pulling, **run `xcodegen generate` first** or you'll build a stale project.

```bash
xcodegen generate                                    # regenerate the project (do this first)
./scripts/gates.sh all                               # full merge gate (tier 0+1+2) — source of truth

# Individual gates:
xcodebuild -scheme RAWCamera -sdk iphoneos -destination 'generic/platform=iOS' \
  CODE_SIGNING_ALLOWED=NO build                       # T0-1 app compiles
xcrun -sdk iphoneos metal -c App/Monitoring/CameraShaders.metal -o /dev/null   # T0-2 shaders
swiftlint --strict --quiet                            # T0-3a
swift-format lint --strict --recursive App AppUITests CameraCore/Sources CameraCore/Tests  # T0-3b
(cd CameraCore && swift test)                         # Tier 1 — CameraCore unit tests

# UI tests (simulator) — drive the real app for hit-testing / layout / overlay regressions:
xcodebuild test -scheme RAWCamera \
  -destination 'platform=iOS Simulator,id=<UDID>'     # iOS 26.x AND 18.x — test both
```

Run the UI suite on **both** an iOS 26.x simulator (Liquid Glass path) and an 18.x one
(fallback path) — they exercise different code.

## Toolchain notes (gotchas that have bitten)

- **Metal toolchain** must be installed: `xcodebuild -downloadComponent MetalToolchain`. Without
  it, `xcrun metal` and any app build with a `.metal` file fail.
- Tools via Homebrew: `swiftlint`, `swift-format`, `xcodegen`. (`status.md` is hand-maintained;
  there is no `ecc` CLI.)
- Xcode 26.x / Swift 6. **Swift 6 strict concurrency is ON** (`SWIFT_STRICT_CONCURRENCY: complete`).
- The project is **iPhone-only** (`TARGETED_DEVICE_FAMILY: "1"`), **portrait-locked**
  (`Info.plist`), and **dark-appearance-locked** (`UIUserInterfaceStyle: Dark`). Don't undo these
  without reason — iPhone-only avoids the iPad all-orientations App Store validation failure.
- **App icon:** an asset-catalog `AppIcon` set rendered from `app-icon.svg`. A Liquid Glass
  Icon Composer `.icon` migration was attempted and abandoned; if `project.yml` ever names an
  `AppIcon` that isn't in the catalog, `actool` fails the build ("no matching app icon named
  AppIcon"). Regenerate the PNG from `app-icon.svg` (1024px, alpha stripped) if it goes missing.

## Conventions

- Push logic into `CameraCore` and unit-test it (test-first). AVFoundation/Metal/SwiftUI stay in
  `App/`. Keep `CameraCore` free of media frameworks.
- Files focused & small (≈200–400 lines, 800 max); many small files over few large ones.
- `let` over `var`; value types by default; immutable updates. Handle errors explicitly — never
  swallow. No networking, no analytics, no secrets (gated by `tier2`).
- Formatting/lint must be clean (`swiftlint --strict`, `swift-format lint --strict`). Config:
  `.swiftlint.yml`, `.swift-format`.
- Frozen surfaces (do not drift without updating CONTRACTS.md on both sides): the `PreviewUniforms`
  ABI (Swift struct + the matching `struct` in `CameraShaders.metal`, guarded by a layout test +
  Metal `static_assert`), the `CameraCapturing` protocol, and the `CameraModel` observable surface.

## Verification reality (read before "it works")

This is a camera app: **live capture and the camera preview cannot be verified in the simulator**
(no camera; RAW capture especially). The honest gate is a human on a device — see
[DEVICE-CHECKLIST.md](DEVICE-CHECKLIST.md). What *is* machine-verifiable: CameraCore logic (unit
tests), compilation, lint, and **UI layout/hit-testing via the simulator UI test harness**
(`AppUITests`), including a screenshot-driven workflow (launch the app under a UI test, screenshot
`app.screenshot()`, export attachments with `xcrun xcresulttool export attachments`). The landscape
UI rotates the simulated device with `XCUIDevice.shared.orientation`. Prefer adding a throwaway
probe test + screenshot to *see* a UI change rather than reasoning about it blind — the UI has
several non-obvious traps documented in **[.claude/docs/gotchas.md](.claude/docs/gotchas.md)**.

## Where things live / design notes

- **Control surface** (`ControlsPanel`): a docked command bar (status · shutter · settings toggle)
  with a slide-up settings drawer (exposure, white balance, focus, ratio, monitoring, format,
  capture). Dark-luxury, Liquid Glass on iOS 26 with a material fallback. UI/design details in
  **[.claude/docs/ui-and-design.md](.claude/docs/ui-and-design.md)**.
- **Monitoring overlays:** histogram, an instrument-style artificial-horizon level guide, zebra &
  focus-peaking (Metal shader), optional zoom slider. Thresholds/colors flow to the Metal shader
  via `PreviewUniforms`.
- **Landscape:** the interface is portrait-locked; floating overlays are counter-rotated to face
  the user. This is a recurring source of layout bugs — see the gotchas doc.
